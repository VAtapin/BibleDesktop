# 2. Общий вывод

Макет показывает не landing page, а рабочий desktop-интерфейс Bible Reader. Это совпадает с целью проекта: первая поверхность должна быть инструментом для чтения и анализа, а не маркетинговой страницей.

---

# 3. Основные зоны интерфейса

## 3.1 Верхняя панель

Содержит:

* логотип Bible Desktop;
* глобальный поиск;
* область уведомлений;
* имя и email пользователя;
* аватар.

Назначение:

* быстрый поиск;
* глобальная навигация;
* пользовательский контекст.

## 3.2 Верхняя рабочая строка

Содержит:

* заголовок текущей группы/режима;
* индикатор уведомлений;
* избранное;
* меню дополнительных действий.

В макете пример:

```text
Группа № 121 (Анализ текста)
```

Вывод:

* группы и анализ текста задуманы как часть основного reader workflow;
* даже если группы уйдут в 2.0, layout должен оставить место для такого режима.

## 3.3 Вкладки чтения

Примеры из макета:

```text
Бытие 4
Откровение 1
Второзаконие 28
```

Особенности:

* вкладки находятся под верхней панелью;
* активная вкладка визуально выделена;
* есть состояние закрытой/заблокированной вкладки.

Вывод:

* модель UI должна поддерживать несколько reader tabs;
* состояние вкладки должно включать перевод, книгу, главу, позицию и правую панель.

## 3.4 Левая вертикальная панель

Иконки предполагают быстрые разделы:

* библиотека/книги;
* закладки;
* история;
* параллельный режим;
* поиск;
* дополнительные инструменты;
* Strong.

Вывод:

* это не sidebar с текстовыми пунктами, а compact tool rail;
* для MVP можно реализовать структуру панели с частью неактивных пунктов, чтобы не ломать layout позже.

## 3.5 Центральная область чтения

Содержит:

* выбор перевода;
* выбор книги/главы;
* ряд toolbar-действий;
* список стихов;
* номера стихов выделены синим;
* нижнюю навигацию назад/вперёд.

Вывод:

* центральный reader должен быть плотным, читаемым и инструментальным;
* стихи должны быть интерактивными объектами, а не просто HTML-текстом;
* номер стиха должен быть отдельной кликабельной зоной.

## 3.6 Правая панель анализа

В макете активен режим:

```text
Заметки
```

Также видны режимы:

```text
Мой перевод
Толкование
```

Панель содержит:

* карточку/сообщение заметки;
* автора;
* время и дату;
* блок событий;
* поле комментария;
* кнопку отправки.

Вывод:

* правая панель является ключевой частью продукта;
* она должна быть контекстной для выбранного стиха/главы/группы;
* в MVP можно начать с заметок и заглушек для остальных режимов.

## 3.7 Нижняя строка

Содержит:

* выбор языка интерфейса;
* ссылки `Информация`, `О проекте`, `Разработчики`, `Контакты`.

Вывод:

* языковое переключение должно быть доступно в основном shell;
* footer не должен занимать много места.

# 4. Приоритеты для MVP UI

Для первого рабочего вертикального среза достаточно:

1. Основной shell с верхней панелью.
2. Вкладки чтения.
3. Левая compact tool rail.
4. Центральное чтение главы.
5. Правая панель с режимом `Заметки`.
6. Выбор перевода, книги и главы.
7. Контекстное действие на стих.

Не обязательно в MVP:

* полный pixel-perfect перенос PSD;
* все режимы правой панели;
* группы;
* события;
* полный набор иконок;
* сложные анимации.

---

# 5. UI-решения для будущей реализации

* Первый экран web-приложения должен открывать reader, а не landing page.
* Layout лучше строить как application shell: topbar + tabs + tool rail + main reader + right panel.
* Правая панель должна быть отдельным состоянием reader tab.
* Вкладки должны храниться на frontend и синхронизироваться с backend для авторизованных пользователей позже.
* Для мобильного интерфейса этот layout нужно трансформировать в bottom navigation + drawer panels, но это не блокер MVP desktop.
